# End-to-End Content Production System

## Context

Today the project is a **voice-cloning pipeline**: a Python CLI (`main.py` → `pipeline/two_stage_runner.py`) that learns a writer's voice from samples and reproduces it on a topic+outline. It only handles the _how it's written_ half — voice — and treats content structure as a user-supplied outline.

We are building an **end-to-end content production system** where **quality content comes first, voice adaptation second**. It is a React + FastAPI app driving four chat-based phases (Ideation → Structure → Drafting → Final Content), maximally transparent: every LLM call/response and every state file is visible live, and all state files are user-editable.

**Hard constraint (user directive):** the existing voice system is a _separate subsystem_ and stays essentially untouched. The new content pipeline **only consumes its output** — the generated voice-prompt text file (`outputs/run_two_stage_*/final_style_prompt.txt`, or the latest `iter_NN_style_prompt.txt` for incomplete runs). Voice profiles are created/regenerated by the existing system on its own; the content app just lists and reads those prompt files. We do **not** refactor `_run_stage2`, the agents, or `llm/client.py`.

## Confirmed decisions

- Backend: FastAPI wrapping existing Python infra; React (Vite + TS) frontend.
- State: flat files per project dir (mirrors `outputs/run_*` convention); directly user-editable in UI.
- Content type chosen **before** Ideation; drives **both** Phase 2 (structural questions) and Phase 3 (detailed draft).
- Phase 4 = **single-shot** generation: selected voice prompt (HOW) + detailed draft (WHAT) → final content. No iterative loop.
- Voice-profile creation = existing separate system, untouched. Content app reads its output files only.
- Streaming token-by-token in chat; explicit "move to next phase" button (no auto-advance); multiple concurrent projects + multiple voice profiles.

## Architecture

Existing dirs (`agents/ prompts/ models/ pipeline/ llm/ main.py config.py outputs/ input/ old-pipeline-skills/`) stay as-is. New code:

```
server/                    # FastAPI package
  app.py settings.py run.py
  llm_stream.py            # async streaming layer (see below)
  domain/{phases,content_types,schemas,events}.py
  storage/{projects,paths,voice_profiles}.py
  phases/{base,prompt_assembly,ideation,structure,drafting,final_content}.py
  activity/log.py          # per-project append-only JSONL + in-mem pub/sub
  api/{projects,files,voices,chat,generate,activity}_router.py
projects/<project_id>/     # runtime data root (gitignored)
web/                       # Vite + React + TS app
```

Per-project dir (`projects/proj_<TS>_<slug>/`): `metadata.json`, `ideation.md`, `structure.md`, `draft.md`, `final_content.md`, `chat_ideation.jsonl`, `chat_structure.jsonl`, `activity.jsonl`.

### Streaming layer — `server/llm_stream.py` (do NOT modify `llm/client.py`)

`llm/client.py` is sync, 2-message-only, and streams only to stdout/stderr — unusable for HTTP token streaming and unsafe under concurrency (module-global logger). Instead add a standalone `StreamingLLM` using `openai.AsyncOpenAI`, **mirroring `LLMClient.__init__`'s env config exactly** (`LLM_PROVIDER`, `OPENAI_BASE_URL`/`OPENCODE_*`, `MODEL_NAME`, `MAX_TOKENS`, reasoning flags → `extra_body`). It takes a full `messages: list[dict]` (real multi-turn, which `LLMClient` can't do) and exposes `async stream(messages, temperature) -> AsyncIterator[{type,text}]` plus a `complete()` convenience. Port `_stream()`'s retry + `finish_reason=='length'` truncation detection so long draft/final generations don't silently truncate. Per-project logging goes to `activity.jsonl`, never the global `enable_detailed_log`.

### API (all under `/api`)

- **Projects** (`projects_router`): POST/GET/GET{id}/PATCH/DELETE `/projects`, plus POST `/projects/{id}/advance` (validates prior phase file exists — the only phase transition; no auto-advance).
- **Files** (`files_router`): GET/PUT `/projects/{id}/files/{name}` (`name ∈ ideation|structure|draft|final_content`); GET `/projects/{id}/transcript/{phase}`. PUT saves user edits and emits `file_changed`.
- **Voices** (`voices_router`): GET `/voices`, GET `/voices/{id}` — scans `outputs/` for `*_style_prompt.txt`; accepts both `final_` and `iter_NN_` (fall back to latest iter when no final).
- **Chat SSE** (`chat_router`): POST `/projects/{id}/phases/{ideation|structure}/chat` `{message}` → event stream: `llm_call` → `token`\* → `assistant_done` → (`llm_call` state-regen) → `file_changed` → `done` (or `error`).
- **Generate SSE** (`generate_router`): POST `/projects/{id}/phases/{drafting|final}/generate` `{instructions?}` → `llm_call` → `token`\* → `file_changed` → `done`.
- **Activity SSE** (`activity_router`): GET `/projects/{id}/activity/stream` (long-lived, all events) + GET `/projects/{id}/activity` (replay history). Per-project `asyncio.Queue` pub/sub in `activity/log.py`.

Chat/generate SSE use POST → frontend reads via `fetch` + stream reader (EventSource can't POST); activity uses native `EventSource`.

### Per-phase prompt composition (`phases/prompt_assembly.py`)

Content-type → framework mapping (`domain/content_types.py`), files read verbatim from `old-pipeline-skills/`:
`linkedin_post → LinkedIn-Post.md + LinkedIn-Framework.md`, `blog_post → Blog-Post.md`, `case_study → Case-Study-Post.md`, `use_case → Use-Case-Post.md`, `article → Article-Framework.md`.

- **Phase 1 Ideation** — 2 calls/turn. (A) reply: system = `Idea-Thinking-Partner-Skill.md` + guardrail "collect CONTEXT/DATA only, never structure; ask 1–2 probing questions"; messages = `chat_ideation.jsonl` + new msg. (B) state-regen: rewrite full `ideation.md` from current on-disk file + transcript (reads current file as base so manual edits survive).
- **Phase 2 Structure** — content-type-aware. (A) system = `Grill-Me.md` + framework file(s) + `ideation.md` as read-only ref + guardrail "ask ONE structural question at a time, dynamic follow-ups, no fixed questionnaire". (B) regenerate `structure.md` conforming to framework skeleton shape.
- **Phase 3 Drafting** — single streamed generation → `draft.md`. system = "produce an EXHAUSTIVELY detailed draft skeleton (message, key points/data, argument beats, transitions, approx length, phrasing anchors; WHAT not final prose; zero voice styling)" + framework file(s); user = `ideation.md` + `structure.md` + optional instructions. Intentionally far more detailed than the old `skeleton_generator`.
- **Phase 4 Final** — single call. system = voice prompt file **verbatim**; user = instruction ("follow the draft's content/order exactly, write flowing final prose, match the system-prompt voice") + `draft.md`. Stream → `final_content.md`. Resolver falls back to latest `iter_NN_style_prompt.txt` if run has no `final_`.

### Frontend (`web/`, Vite + React + TS)

`@tanstack/react-query` for REST; `fetch`+ReadableStream for chat/generate SSE; `EventSource` for activity; `react-markdown` for rendering; textarea/CodeMirror for editing. Tree: `AppLayout` → `ProjectSidebar` (list + `NewProjectDialog` with title/content-type/voice), `ProjectWorkspace` (`ProjectHeader` + `VoiceProfilePicker`, `PhaseStepper` with "Move to next phase", `PhasePanel` → `IdeationChat`/`StructureChat` reusing `ChatPane`, `DraftingPanel`, `FinalPanel`), `StateFileViewer` (tabs + view/edit, live via `file_changed`), `ActivityLogPane` (live LLM feed). Hooks: `useProjects/useProject/useCreateProject/useAdvancePhase/useVoices/useFile/useSaveFile/useTranscript/useChatStream/useGenerateStream/useActivityStream`; `lib/sse.ts` parses SSE frames.

- **New:** everything under `server/` and `web/` (above).
- **Modify:** `requirements.txt` (pin `fastapi`, `uvicorn`, `sse-starlette`, `anyio`, add `openai` async is same pkg); `.gitignore` (add `projects/`).
- **Untouched:** all voice-pipeline dirs, `llm/client.py`, `main.py`, `config.py` (reused read-only for env/dotenv conventions).

## Build order

1. Backend skeleton: settings/app/storage/domain + projects/files/voices routers. Verify via `curl` (create project, read/write file, list voices).
2. `server/llm_stream.py` with config parity + retry/truncation. Verify with a scratch streaming route.
3. Activity bus + Phase 1/2 chat (`ideation`, `structure`) + chat/activity routers. Verify `curl -N` raw SSE updates `ideation.md`/`structure.md` and emits `file_changed`.
4. Phase 3/4 generation + generate router. Verify detailed `draft.md`; verify final uses selected voice file as system prompt.
5. Frontend: sidebar + project creation + file viewer (REST) → ChatPane streaming → activity pane → drafting/final buttons.
6. Polish: phase gating in `/advance`, status badges, error toasts, voice iter-file fallback, gate composer while streaming.

## Verification (end-to-end)

1. Backend: `python -m server.run` (uvicorn `:8000`, env from existing `.env`).
2. Frontend: `cd web && npm install && npm run dev` (Vite `:5173`, proxy `/api` → `:8000`).
3. Manual walk-through: create LinkedIn project + pick a voice from the `outputs/`-scanned dropdown → Ideation (tokens stream, `ideation.md` updates live, activity shows call+file event, stays context-only) → advance → Structure (content-type-aware questions one at a time, `structure.md` builds) → advance → Drafting (Generate streams a richly detailed `draft.md`; hand-edit + Save persists) → advance → Final (one-shot `final_content.md` in the selected voice, activity shows single call whose system prompt is the voice file) → reload page, confirm everything rehydrates from `projects/<id>/`.

## Risks

- Sync client concurrency → bypassed by `AsyncOpenAI` in `llm_stream.py`; sync `LLMClient` never called from handlers.
- Module-global detailed logger in `client.py` → not reused; per-project `activity.jsonl` instead.
- No multi-turn in `LLMClient` → server owns `messages` history in JSONL.
- Context-length on Phase 3/4 (`draft.md` + voice prompt may exceed `MAX_TOKENS`) → surface `event: error`, don't truncate silently.
- User edit vs regen race → Call B reads current on-disk file as base; gate composer during streaming.

# Clean Implementation Plan

This section turns the product plan into an implementation blueprint. It keeps the existing voice-cloning pipeline as a read-only subsystem and builds the new content app beside it.

## Guiding constraints

1. **No voice-pipeline refactor.** Do not modify `main.py`, `config.py`, `pipeline/`, `agents/`, `models/`, `prompts/`, or `llm/client.py` unless a later bug requires a tiny compatibility fix. The content app reads voice prompt artifacts only.
2. **Flat files are the source of truth.** Runtime state lives in `projects/<project_id>/`; the frontend should rehydrate from disk after reload with no hidden in-memory-only state.
3. **Every generation is inspectable.** Each LLM call writes a full activity event with request metadata, prompt/messages, streamed response text, finish reason, and errors.
4. **Phase movement is explicit.** The server never advances a project automatically. `/advance` is the only state transition endpoint.
5. **Prompt assets are read verbatim.** Phase prompts compose existing markdown files from `Old-pipeline-skills/`; add path resolution that also accepts the lowercase spelling used in the architecture notes.
6. **HTTP handlers stay thin.** Routers validate and stream; storage, prompt assembly, LLM streaming, and phase behavior live in separate modules.

## Target module contracts

Responsibilities:

- Load `.env` by importing existing `config` for side effects.
- Define repo-root-relative paths:
  - `ROOT_DIR`
  - `OUTPUTS_DIR = ROOT_DIR / "outputs"`
  - `PROJECTS_DIR = ROOT_DIR / "projects"`
  - `PROMPT_SKILLS_DIR`, resolved by checking `Old-pipeline-skills` then `old-pipeline-skills`
- Define API/runtime settings:
  - `API_HOST = "127.0.0.1"`
  - `API_PORT = 8000`
  - `CORS_ORIGINS = ["http://localhost:5173", "http://127.0.0.1:5173"]`

This module should not construct OpenAI clients; it only centralizes paths and simple constants.

### `server/domain/phases.py`

Use enums instead of loose strings:

- `Phase = ideation | structure | drafting | final`
- `PhaseFile = ideation | structure | draft | final_content`

Define the ordered phase list and helpers:

- `next_phase(phase) -> Phase | None`
- `required_file_for_advance(current_phase) -> PhaseFile`
- `is_chat_phase(phase)`, `is_generate_phase(phase)`

### `server/domain/content_types.py`

Define `ContentType` enum and a framework mapping:

- `linkedin_post`: `3-Drafting/LinkedIn-Post.md`, `3-Drafting/LinkedIn-Framework.md`
- `blog_post`: `3-Drafting/Blog-Post.md`
- `case_study`: `3-Drafting/Case-Study-Post.md`
- `use_case`: `3-Drafting/Use-Case-Post.md`
- `article`: `3-Drafting/Article-Framework.md`

Expose `framework_files_for(content_type) -> list[Path]`. Fail fast with a clear error if a configured markdown file is missing.

### `server/domain/schemas.py`

Use Pydantic models for API input/output and persisted metadata.

Core models:

- `ProjectCreate`: `title`, `content_type`, optional `voice_id`
- `ProjectPatch`: optional `title`, `content_type`, `voice_id`
- `ProjectMetadata`: `id`, `title`, `slug`, `content_type`, `phase`, `voice_id`, `created_at`, `updated_at`
- `ProjectSummary`: metadata plus booleans for existing state files
- `FilePayload`: `name`, `content`, `updated_at`
- `VoiceProfile`: `id`, `run_id`, `label`, `path`, `source`, `iteration`, `mtime`, `preview`
- `ChatRequest`: `message`
- `GenerateRequest`: optional `instructions`, optional `voice_id`
- `ActivityEvent`: `id`, `ts`, `project_id`, `type`, `phase`, `payload`

Keep event payloads typed enough for server correctness, but flexible enough for LLM-specific metadata.

### `server/domain/events.py`

Define event type constants:

- `project_created`
- `project_updated`
- `phase_advanced`
- `llm_call`
- `token`
- `assistant_done`
- `file_changed`
- `error`
- `done`

Use these same strings for SSE events and `activity.jsonl` entries.

## Storage layer

### `server/storage/paths.py`

Responsibilities:

- Validate project ids with a strict regex: `^proj_[0-9]{8}_[0-9]{6}_[a-z0-9-]+$`.
- Validate public file names through enum mapping only:
  - `ideation -> ideation.md`
  - `structure -> structure.md`
  - `draft -> draft.md`
  - `final_content -> final_content.md`
- Create project directories.
- Provide `atomic_write_text(path, text)` using a temp file in the same directory and `Path.replace()`.
- Never join unchecked user input into a path.

### `server/storage/projects.py`

Responsibilities:

- `create_project(title, content_type, voice_id) -> ProjectMetadata`
  - Generate `proj_<YYYYMMDD>_<HHMMSS>_<slug>`.
  - Create empty state files only when useful for UX; otherwise lazy-create on first write. Prefer lazy state files so “advance” can validate meaningful content.
  - Write `metadata.json` atomically.
  - Initialize `activity.jsonl`.
- `list_projects() -> list[ProjectSummary]`
- `get_project(project_id) -> ProjectMetadata`
- `patch_project(project_id, patch) -> ProjectMetadata`
- `delete_project(project_id) -> None`
- `advance_project(project_id) -> ProjectMetadata`
  - Validate required prior phase file exists and is non-empty.
  - Preserve final phase when no next phase exists.

Implementation detail: use one `asyncio.Lock` per project id for chat/generate/file writes. Store locks in a process-local map; flat files remain authoritative across restart.

### `server/storage/voice_profiles.py`

Responsibilities:

- Scan `outputs/run_two_stage_*` only.
- Prefer `final_style_prompt.txt` when present.
- If no final prompt exists, select the highest numbered `iter_NN_style_prompt.txt`.
- Return one `VoiceProfile` per run.
- Build stable ids from the relative prompt path, for example URL-safe base64 or a slugified `run_id:filename`.
- Resolve `GET /voices/{id}` only from the scan index; do not accept arbitrary paths.
- Include a short preview from the prompt file for the picker.

Keep QNA `final_prompt.txt` out of scope unless the product plan later explicitly allows it.

## Activity and SSE

### `server/activity/log.py`

Build one event API that all routes and phases use:

- `append_event(project_id, event_type, phase, payload) -> ActivityEvent`
- `history(project_id, after_id=None) -> list[ActivityEvent]`
- `subscribe(project_id) -> AsyncIterator[ActivityEvent]`
- `publish(event)`

Persistence:

- Append compact JSON lines to `projects/<id>/activity.jsonl`.
- Add monotonically increasing ids per project. Use `time.time_ns()` plus a short counter or derive from line count under the project lock.
- Store full prompt/messages and assistant responses on `llm_call` / `assistant_done`, but never store API keys or client internals.

Pub/sub:

- Keep process-local subscriber queues.
- On subscribe, create an `asyncio.Queue[ActivityEvent]`, yield new events until disconnect, then remove the queue.
- Bound queue size and drop with an explicit error event if a client is too slow.

SSE formatting:

- Use a shared helper to emit:
  - `event: <type>`
  - `id: <event.id>`
  - `data: <json>`
- `chat` and `generate` endpoints return POST streams with the same frame format.
- `activity/stream` can use `EventSourceResponse` or `StreamingResponse`; keep the wire format identical.

## LLM streaming layer

### `server/llm_stream.py`

Create `StreamingLLM`; do not import or wrap `LLMClient`.

Constructor parity with `LLMClient.__init__`:

- Provider selection:
  - `LLM_PROVIDER=opencode` uses `OPENCODE_BASE_URL`, `OPENCODE_API_KEY`, default model `deepseek-v4-flash-free`.
  - Otherwise use `OPENAI_BASE_URL`, `OPENAI_API_KEY`, default model `z-ai/glm-5.1`.
- Shared env:
  - `MODEL_NAME`
  - `MAX_TOKENS`
  - `ENABLE_THINKING`
  - `REASONING_EFFORT`
  - `REASONING_BUDGET`
  - `SHOW_REASONING`
- Build the same `extra_body` rules as `LLMClient`.
- On a `BadRequestError` mentioning `chat_template`, retry once with thinking disabled for this instance.

Methods:

- `stream(messages: list[dict], temperature: float, label: str) -> AsyncIterator[LLMStreamEvent]`
- `complete(messages, temperature, label) -> str`

Stream event shape:

- `{"type": "token", "text": "..."}`
- `{"type": "reasoning", "text": "..."}` only if `SHOW_REASONING=true`; the UI can hide it by default.
- `{"type": "done", "finish_reason": "stop", "text": full_response}`

Reliability:

- Retry generic connection failures up to the existing six-attempt policy.
- Use exponential backoff for `RateLimitError` matching the old client behavior.
- If `finish_reason == "length"`, raise `TruncatedResponseError(partial_text)`.
- Routes should surface truncation as an SSE `error` event and should not write partial state files.

## Prompt assembly and phases

### `server/phases/prompt_assembly.py`

Provide small pure functions that return `list[dict[str, str]]` messages.

Helpers:

- `read_skill(relative_path) -> str`
- `read_frameworks(content_type) -> str`
- `read_project_file(project_id, phase_file) -> str`
- `load_transcript(project_id, phase) -> list[dict]`
- `conversation_messages(system, transcript, new_user_message) -> list[dict]`

Keep prompts explicit and testable. Avoid hidden global prompt strings outside this module.

### `server/phases/ideation.py`

Flow for `chat(project, message)`:

1. Acquire the project lock.
2. Append the user message to `chat_ideation.jsonl`.
3. Call A:
   - System = `1-Ideation/Idea-Thinking-Partner-Skill.md` plus guardrails:
     - collect context/data only
     - do not structure the piece
     - ask one or two probing questions
   - Messages = full ideation transcript including the new user message.
4. Stream assistant tokens to the POST SSE response and activity log.
5. Append assistant response to `chat_ideation.jsonl`.
6. Call B state regeneration:
   - System = deterministic state writer.
   - User = current on-disk `ideation.md` plus full transcript.
   - Requirement: preserve and integrate manual edits unless directly contradicted.
7. Write `ideation.md` atomically.
8. Emit `file_changed`, then `done`.

Temperature:

- Call A: `0.7`
- Call B: `0.2`

### `server/phases/structure.py`

Flow mirrors ideation with different prompts:

- System = `2-Positioning/Grill-Me.md` plus content-type framework files plus guardrails:
  - use `ideation.md` as read-only source material
  - ask one structural question at a time
  - make dynamic follow-ups
  - do not run a fixed questionnaire
- State regen writes `structure.md` in the shape implied by the selected framework files.

Temperature:

- Chat reply: `0.7`
- State regen: `0.2`

### `server/phases/drafting.py`

Flow for `generate(project, instructions)`:

1. Lock project.
2. Validate `ideation.md` and `structure.md` are non-empty.
3. Build messages:
   - System = drafting instruction plus framework files.
   - User = `ideation.md`, `structure.md`, optional user instructions.
4. Stream tokens to client and activity.
5. Emit `assistant_done` with the full response text.
6. On successful completion, write `draft.md`.
7. Emit `file_changed`, then `done`.

The draft is not final prose. It should be dense, explicit, and voice-neutral: content order, argument beats, examples/data, transitions, length guidance, and phrasing anchors.

### `server/phases/final_content.py`

Flow for `generate(project, instructions)`:

1. Lock project.
2. Validate `draft.md` is non-empty.
3. Resolve voice id:
   - request `voice_id` overrides project `voice_id`
   - project `voice_id` is required if not supplied
4. Read the selected voice prompt file verbatim.
5. Build messages:
   - System = voice prompt file text.
   - User = instruction to follow the draft content/order exactly, optional user instructions, then `draft.md`.
6. Stream tokens.
7. Emit `assistant_done` with the full response text.
8. On success, write `final_content.md`.
9. Emit `file_changed`, then `done`.

Do not call the old content generator, style extractor, comparator, or refiner.

  - `file_changed`

### `server/api/activity_router.py`

Endpoints:

- `GET /api/projects/{project_id}/activity`
- `GET /api/projects/{project_id}/activity/stream`

Behavior:

- History endpoint returns persisted JSONL.

### Frontend state rules

- React Query owns REST-loaded project/file/voice data.
- Streaming hooks own transient token buffers.
- On `file_changed`, invalidate the matching file query and project list.
- Disable chat/generate controls while a stream is active for that project.
- Keep manual file edits local until Save; warn before switching tabs with dirty edits.
- Show server errors from SSE `error` events as visible toasts or inline banners.

### SSE parser

`lib/sse.ts` should parse frames from `fetch(...).body.getReader()`:

- Split by blank lines.
- Support `event:`, `id:`, and multi-line `data:`.
- JSON-parse `data`.
- Call `onEvent({ event, id, data })`.

Use native `EventSource` only for the long-lived activity stream.

### UI style

Keep the first screen as the actual working app, not a landing page.

Layout:

- Left sidebar: projects.
- Center: phase work surface.
- Right panel: activity log.
- Lower or side tab area: editable state files.

Style direction:

- Dense, operational, quiet interface.
- Clear phase status and disabled states.
- Monospace only for prompts/raw files/activity payloads.
- Avoid decorative hero sections, nested cards, and oversized marketing typography.

## Verification strategy

### Automated backend tests

Add a small `tests/` suite once the backend exists:

- Storage:
  - project ids validate
  - path traversal is rejected
  - file writes are atomic
  - advance fails on missing files and succeeds on non-empty required files
- Voice scanner:
  - final prompt preferred over iter prompt
  - latest `iter_NN_style_prompt.txt` selected when final is absent
  - arbitrary ids cannot resolve paths
- Prompt assembly:
  - each content type loads expected framework files
  - missing skill files fail clearly
- Phases:
  - fake `StreamingLLM` produces deterministic tokens
  - ideation chat appends transcript and writes `ideation.md`
  - drafting does not write `draft.md` on truncation
- SSE parser/event order:
  - chat event order matches contract
  - generate event order matches contract

Keep LLM-network tests out of the default suite.

## Implementation order inside each slice

Use this order to keep changes reviewable:

1. Define schemas/enums first.
2. Implement pure path/storage/prompt functions.
3. Add routers around those functions.
4. Add streaming and side effects.
5. Add UI over already-working API surfaces.
6. Add tests around the behavior that could silently corrupt state.
7. Run manual end-to-end verification.

## Cleanliness checklist

- No route directly reads arbitrary user-supplied paths.
- No old voice-pipeline modules are imported by the content phases.
- No LLM call is made without an activity `llm_call` record.
- No partial state file is written after an LLM error or truncation.
- No phase transition happens outside `/advance`.
- No frontend component owns server truth that should live in React Query.
- No hidden prompt text is scattered through routers or components.
- No `.env`, `projects/`, `outputs/`, or generated dependency folders are committed.
