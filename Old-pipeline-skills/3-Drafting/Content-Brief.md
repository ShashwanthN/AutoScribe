# Content Brief Reference

This file defines the strategic fields every content type should resolve before drafting.

The content app has four phases:

1. Ideation captures raw material.
2. Structure decides the argument or narrative shape.
3. Drafting creates an exhaustive WHAT-only draft.
4. Final Content applies the selected voice prompt.

This brief belongs to phases 2 and 3. It should never introduce voice styling.

## Required Fields

```markdown
# Content Brief

## Working Title
[Plain working title, not final headline.]

## Content Type
[LinkedIn post / blog post / case study / use case / article.]

## Target Reader
[Specific role, context, maturity, and current belief.]

## Reader Shift
Before reading, the reader believes:

After reading, the reader should believe:

## Core Message
[One sentence. If it needs multiple independent claims, narrow it.]

## Why Now
[What makes this timely, costly, or newly visible?]

## Evidence Bank
- [Story, example, data point, observation, customer fact, operational detail.]

## Counterargument
[The strongest smart objection and how the piece should handle it.]

## Desired Action
[What the reader should do, think, question, save, share, or discuss.]

## Constraints
[Length, platform, claims to avoid, facts that must be included, facts that must not be invented.]
```

## Quality Bar

A strong brief is:

- specific enough to draft from
- narrow enough to remember
- supported by concrete material
- honest about uncertainty
- clear on what the reader gets
- free of voice instructions

## Weak Brief Signals

- generic audience labels like "leaders" or "businesses"
- core message with multiple unrelated claims
- no proof beyond opinion
- no reader before/after shift
- no clear ending action
- solution language before the problem is understood
